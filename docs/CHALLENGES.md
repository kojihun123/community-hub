# CommunityHub Challenges

## Overview

이 문서는 CommunityHub를 구현하면서 실제로 부딪힌 문제와 해결 과정을 간단히 정리한 기록이다.
기능 목록만 나열하는 대신, 설계와 구현 과정에서 어떤 고민이 있었는지를 남기는 목적이다.

## 1. Blade 구조를 어디까지 나눌지

초기에는 `app.blade.php`, 네비게이션, 푸터, 홈 화면이 한 파일 안에 많이 모여 있었다.
반복되는 UI를 무조건 잘게 나누면 오히려 관리가 어려워질 수 있어서, 실제로 반복되는 단위만 분리하는 기준이 필요했다.

정리 과정:

- 레이아웃은 `layouts/`
- 재사용 UI는 `components/`
- 페이지 일부 조각은 `partials/`

로 구분했다.

특히 `제목 + 우측 액션 + 본문` 구조는 반복 사용 빈도가 높아서 `x-ui.section-card`로 공통화했다.
반대로 아직 한두 번밖에 안 쓰는 마크업은 일부러 과하게 쪼개지 않았다.

## 2. `flex`와 `grid` 선택

홈 화면과 게시판 목록을 만들 때 `flex`로 처리할지 `grid`로 처리할지가 애매한 경우가 많았다.
특히 긴 제목과 좋아요 수를 같은 줄에 배치할 때, 단순 `flex`만으로는 제목이 밀리거나 말줄임이 어색해지는 문제가 있었다.

정리 기준:

- 정렬 중심: `flex`
- 칸 비율과 레이아웃 분배: `grid`

이 기준으로 정리하면서 홈 카드와 게시판 목록 화면이 훨씬 안정됐다.

## 3. Tailwind 반영이 안 되는 것처럼 보였던 문제

`grid-cols-3` 같은 클래스를 분명히 넣었는데도 화면이 한 줄 3칸으로 보이지 않는 문제가 있었다.
결과적으로는 Blade 코드 문제라기보다, Vite/Tailwind 빌드가 최신 상태로 반영되지 않았던 문제였다.

이 경험을 통해:

- Blade 클래스가 맞는지 확인
- 반응형 브레이크포인트인지 확인
- `npm run dev` 또는 `npm run build` 상태 확인

이 순서로 보는 습관이 필요하다는 점을 정리했다.

## 4. 게시판 URL 구조 재설계

처음에는 게시판 목록은 `/boards/{slug}`인데 게시글 상세는 `/posts/{id}`라서 URL 흐름이 끊겨 보였다.
커뮤니티 서비스 느낌을 살리기 위해 게시글 상세를 게시판 흐름 안으로 넣는 쪽이 더 자연스럽다고 판단했다.

변경 후 구조:

- `/boards/{slug}`
- `/boards/{board:slug}/posts/{post}`
- `/boards/{board:slug}/posts/create`

이 과정에서 Laravel의 route model binding과 라우트 선언 순서가 중요하다는 점도 같이 확인했다.
특히 `create` 같은 고정 문자열 라우트는 `{post}`보다 먼저 선언해야 했다.

## 5. `FormRequest`에서 `content`가 이상하게 들어오던 문제

게시글 저장 시 `content` 값에 HTML 본문이 아니라 전체 폼 쿼리 문자열이 들어오는 문제가 있었다.
처음에는 CKEditor 문제처럼 보였지만, 실제 원인은 `StorePostRequest`의 `prepareForValidation()`이었다.

문제 코드:

```php
$this->content
```

해결 방식:

```php
$this->input('content')
```

처럼 명시적으로 input 값을 읽도록 바꿨다.

이 경험으로 `FormRequest` 전처리에서는 `$this->input()`을 사용하는 쪽이 더 안전하다는 점을 확인했다.

## 6. CKEditor 도입 방식 변경

초기에는 CDN, 다운로드 파일, UMD 방식 등을 시도했지만 설정 충돌과 라이선스 에러, 초기화 실패가 반복됐다.
결국 현재 프로젝트에서는 `npm + Vite` 방식으로 정리하는 것이 가장 안정적이었다.

정리 포인트:

- Blade 안에서 에디터 전체를 처리하지 않고
- `resources/js/post-editor.js`로 분리
- `app.js`에서 import
- `hidden input + editor div` 구조로 동기화

이 방식으로 유지보수성이 훨씬 좋아졌다.

## 7. CKEditor 이미지 업로드

처음에는 `Base64UploadAdapter`를 사용했지만, 이 방식은 본문 HTML이 무거워지고 실제 서비스 구조와도 거리가 있었다.
결국 서버 업로드 방식으로 바꾸면서 아래 문제를 함께 해결해야 했다.

- CSRF 토큰 전달
- 업로드 라우트 구성
- 파일 검증
- Storage 저장
- 업로드 실패 시 에러 응답 형식
- 업로드 속도 제한

추가로 `public/storage` 심볼릭 링크가 없어서 이미지 URL이 `403`처럼 보이는 문제도 있었다.
`php artisan storage:link`로 해결했다.

## 8. 첨부파일을 DB에 기록할지 여부

이미지가 본문 안에 들어가기만 하면 되는 것처럼 보였지만, 실제로는 업로드 파일의 추적과 정리가 필요했다.
그래서 `attachments` 테이블을 두는 쪽으로 방향을 정리했다.

이 결정의 이유:

- 어떤 파일이 어떤 글에 속하는지 추적 가능
- 원본 파일명, 경로, MIME 타입, 크기 관리 가능
- 임시 업로드 파일 정리 가능
- 이미지와 일반 파일을 같은 구조로 확장 가능

## 9. 글 작성 중 업로드하고 저장하지 않는 경우

에디터 업로드는 글 저장보다 먼저 일어나기 때문에,
사용자가 이미지를 올리고 글을 끝까지 작성하지 않으면 파일이 고아 데이터가 되는 문제가 있었다.

현재 정리한 방향:

- 업로드 시 `attachments.is_temporary = true`
- 글 저장 시 본문에 실제로 사용된 첨부만 `post_id` 연결
- 연결된 첨부는 `is_temporary = false`
- 남은 임시 첨부는 나중에 정리

이 구조는 기능, 운영, 보안 측면에서 가장 균형이 좋다고 판단했다.

## 10. 업로드 남용 방지

이미지 업로드 기능은 악의적으로 남용될 수 있기 때문에, 단순 구현 이상으로 운영 관점이 필요했다.

현재 반영하거나 고려한 방어 요소:

- `auth`로 로그인 사용자만 업로드 허용
- 파일 크기 제한
- 파일 타입 제한
- 업로드 라우트에 `throttle`
- 예외 발생 시 저장 파일 롤백
- 미사용 임시 첨부 정리 정책

## 11. 게시글 저장과 첨부 연결의 일관성

게시글은 저장됐는데 첨부 연결이 실패하거나,
반대로 첨부만 남는 문제를 막기 위해 `PostController@store()`에 트랜잭션을 적용했다.

정리된 방향:

- 게시글 생성
- 본문 내 이미지 경로 추출
- 현재 사용자 임시 첨부 연결

을 하나의 트랜잭션 안에서 처리하도록 맞췄다.

## 12. 현재까지 얻은 정리

지금까지 구현하면서 가장 크게 정리된 기준은 아래와 같다.

- 반복되는 UI만 공통화한다.
- URL은 커뮤니티 흐름이 보이도록 설계한다.
- 에디터는 Blade보다 JS 엔트리로 분리한다.
- 파일 업로드는 Storage + DB 기록 + 임시 상태 관리까지 함께 본다.
- 실무적으로 중요한 부분은 `예외 처리`, `트랜잭션`, `throttle`, `정리 정책`이다.

이 문서는 이후 기능을 구현하면서 계속 보완할 수 있다.

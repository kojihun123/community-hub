# CommunityHub Challenges

## Overview

이 문서는 CommunityHub를 구현하면서 실제로 부딪힌 문제와 해결 과정을 간단히 정리한 기록이다.
기능 목록만 나열하는 대신, 설계와 구현 과정에서 어떤 고민이 있었는지를 남기는 목적이다.

## 1. Blade 구조를 어디까지 나눌지

초기에는 `app.blade.php`, 네비게이션, 푸터, 홈 화면이 한 파일 안에 많이 모여 있었다.
반복되는 UI를 무조건 잘게 나누면 오히려 관리가 어려워질 수 있어서, 실제로 반복되는 단위만 분리하는 기준이 필요했다.

정리 과정:

- 레이아웃은 `layouts/`
- 재사용 UI는 `components/`
- 페이지 일부 조각은 `partials/`

로 구분했다.

특히 `제목 + 우측 액션 + 본문` 구조는 반복 사용 빈도가 높아서 `x-ui.section-card`로 공통화했다.
반대로 아직 한두 번밖에 안 쓰는 마크업은 일부러 과하게 쪼개지 않았다.

## 2. `flex`와 `grid` 선택

홈 화면과 게시판 목록을 만들 때 `flex`로 처리할지 `grid`로 처리할지가 애매한 경우가 많았다.
특히 긴 제목과 좋아요 수를 같은 줄에 배치할 때, 단순 `flex`만으로는 제목이 밀리거나 말줄임이 어색해지는 문제가 있었다.

정리 기준:

- 정렬 중심: `flex`
- 칸 비율과 레이아웃 분배: `grid`

이 기준으로 정리하면서 홈 카드와 게시판 목록 화면이 훨씬 안정됐다.

## 3. Tailwind 반영이 안 되는 것처럼 보였던 문제

`grid-cols-3` 같은 클래스를 분명히 넣었는데도 화면이 한 줄 3칸으로 보이지 않는 문제가 있었다.
결과적으로는 Blade 코드 문제라기보다, Vite/Tailwind 빌드가 최신 상태로 반영되지 않았던 문제였다.

이 경험을 통해:

- Blade 클래스가 맞는지 확인
- 반응형 브레이크포인트인지 확인
- `npm run dev` 또는 `npm run build` 상태 확인

이 순서로 보는 습관이 필요하다는 점을 정리했다.

## 4. 게시판 URL 구조 재설계

처음에는 게시판 목록은 `/boards/{slug}`인데 게시글 상세는 `/posts/{id}`라서 URL 흐름이 끊겨 보였다.
커뮤니티 서비스 느낌을 살리기 위해 게시글 상세를 게시판 흐름 안으로 넣는 쪽이 더 자연스럽다고 판단했다.

변경 후 구조:

- `/boards/{slug}`
- `/boards/{board:slug}/{post}`
- `/boards/{board:slug}/create`

이 과정에서 Laravel의 route model binding과 라우트 선언 순서가 중요하다는 점도 같이 확인했다.
특히 `create` 같은 고정 문자열 라우트는 `{post}`보다 먼저 선언해야 했다.

## 5. `FormRequest`에서 `content`가 이상하게 들어오던 문제

게시글 저장 시 `content` 값에 HTML 본문이 아니라 전체 폼 쿼리 문자열이 들어오는 문제가 있었다.
처음에는 CKEditor 문제처럼 보였지만, 실제 원인은 `PostRequest`의 `prepareForValidation()`이었다.

문제 코드:

```php
$this->content
```

해결 방식:

```php
$this->input('content')
```

처럼 명시적으로 input 값을 읽도록 바꿨다.

이 경험으로 `FormRequest` 전처리에서는 `$this->input()`을 사용하는 쪽이 더 안전하다는 점을 확인했다.

## 6. CKEditor 도입 방식 변경

초기에는 CDN, 다운로드 파일, UMD 방식 등을 시도했지만 설정 충돌과 라이선스 에러, 초기화 실패가 반복됐다.
결국 현재 프로젝트에서는 `npm + Vite` 방식으로 정리하는 것이 가장 안정적이었다.

정리 포인트:

- Blade 안에서 에디터 전체를 처리하지 않고
- `resources/js/post-editor.js`로 분리
- `app.js`에서 import
- `hidden input + editor div` 구조로 동기화

이 방식으로 유지보수성이 훨씬 좋아졌다.

## 7. CKEditor 이미지 업로드

처음에는 `Base64UploadAdapter`를 사용했지만, 이 방식은 본문 HTML이 무거워지고 실제 서비스 구조와도 거리가 있었다.
결국 서버 업로드 방식으로 바꾸면서 아래 문제를 함께 해결해야 했다.

- CSRF 토큰 전달
- 업로드 라우트 구성
- 파일 검증
- Storage 저장
- 업로드 실패 시 에러 응답 형식
- 업로드 속도 제한

추가로 `public/storage` 심볼릭 링크가 없어서 이미지 URL이 `403`처럼 보이는 문제도 있었다.
`php artisan storage:link`로 해결했다.

## 8. 첨부파일을 DB에 기록할지 여부

이미지가 본문 안에 들어가기만 하면 되는 것처럼 보였지만, 실제로는 업로드 파일의 추적과 정리가 필요했다.
그래서 `attachments` 테이블을 두는 쪽으로 방향을 정리했다.

이 결정의 이유:

- 어떤 파일이 어떤 글에 속하는지 추적 가능
- 원본 파일명, 경로, MIME 타입, 크기 관리 가능
- 임시 업로드 파일 정리 가능
- 이미지와 일반 파일을 같은 구조로 확장 가능

## 9. 글 작성 중 업로드하고 저장하지 않는 경우

에디터 업로드는 글 저장보다 먼저 일어나기 때문에,
사용자가 이미지를 올리고 글을 끝까지 작성하지 않으면 파일이 고아 데이터가 되는 문제가 있었다.

현재 정리한 방향:

- 업로드 시 `attachments.is_temporary = true`
- 글 저장 시 본문에 실제로 사용된 첨부만 `post_id` 연결
- 연결된 첨부는 `is_temporary = false`
- 남은 임시 첨부는 나중에 정리

이 구조는 기능, 운영, 보안 측면에서 가장 균형이 좋다고 판단했다.

## 10. 업로드 남용 방지

이미지 업로드 기능은 악의적으로 남용될 수 있기 때문에, 단순 구현 이상으로 운영 관점이 필요했다.

현재 반영하거나 고려한 방어 요소:

- `auth`로 로그인 사용자만 업로드 허용
- 파일 크기 제한
- 파일 타입 제한
- 업로드 라우트에 `throttle`
- 예외 발생 시 저장 파일 롤백
- 미사용 임시 첨부 정리 정책

## 11. 게시글 저장과 첨부 연결의 일관성

게시글은 저장됐는데 첨부 연결이 실패하거나,
반대로 첨부만 남는 문제를 막기 위해 `PostController@store()`에 트랜잭션을 적용했다.

정리된 방향:

- 게시글 생성
- 본문 내 이미지 경로 추출
- 현재 사용자 임시 첨부 연결

을 하나의 트랜잭션 안에서 처리하도록 맞췄다.

## 12. 현재까지 얻은 정리

지금까지 구현하면서 가장 크게 정리된 기준은 아래와 같다.

- 반복되는 UI만 공통화한다.
- URL은 커뮤니티 흐름이 보이도록 설계한다.
- 에디터는 Blade보다 JS 엔트리로 분리한다.
- 파일 업로드는 Storage + DB 기록 + 임시 상태 관리까지 함께 본다.
- 실무적으로 중요한 부분은 `예외 처리`, `트랜잭션`, `throttle`, `정리 정책`이다.

## 13. 중첩 라우트와 `scopeBindings()`

게시글 상세, 수정, 삭제 라우트를 게시판 흐름 안으로 넣으면서
`/boards/{board:slug}/{post}` 같은 중첩 라우트를 사용하게 되었다.

처음에는 컨트롤러에서 아래 같은 검증을 직접 했다.

```php
abort_unless($post->board_id === $board->id, 404);
```

이 방식도 동작은 하지만, 컨트롤러마다 같은 검증이 반복되고
Laravel이 제공하는 라우트 바인딩 기능을 충분히 활용하지 못하는 구조였다.

이후 `scopeBindings()`를 적용하면서,
라우터가 `board`에 속한 `post`만 매칭하도록 정리할 수 있다는 점을 배웠다.

정리된 장점:

- 컨트롤러에서 중복 관계 검증 코드 제거
- 라우터가 소속 관계를 먼저 보장
- 컨트롤러는 상태 확인과 권한 확인에 집중 가능
- 중첩 리소스 구조가 더 Laravel스럽고 읽기 쉬워짐

이 경험을 통해, 단순히 동작하는 코드보다
`Laravel이 원래 제공하는 기능으로 더 자연스럽게 표현하는 방식`을 찾는 것이 중요하다는 점을 정리했다.

## 14. 댓글 수정 폼을 다시 여는 UX

댓글 수정은 Alpine으로 인라인 편집을 적용했기 때문에,
validation 실패 후 페이지가 다시 로드되면 어떤 댓글을 수정 중이었는지 상태가 사라지는 문제가 있었다.

처음에는 모든 댓글의 `editing` 상태가 다시 `false`가 되면서,
에러는 발생했지만 실제 수정 폼은 닫혀버리는 어색한 UX가 나왔다.

해결 방식:

- 수정 폼마다 `editing_comment_id` hidden input 추가
- validation 실패 후 `old('editing_comment_id')` 값을 다시 읽기
- 해당 댓글의 `x-data` 초기값만 `editing: true`로 설정

이 방식으로 "에러가 난 바로 그 댓글"만 다시 수정 상태로 열 수 있게 되었다.

## 15. 댓글 작성 / 수정 / 답글 폼의 `content` 충돌

게시글 상세 화면 안에는

- 댓글 작성 폼
- 댓글 수정 폼
- 답글 작성 폼

이 동시에 존재한다.

처음에는 모든 폼이 `content`라는 같은 필드명을 사용했기 때문에,
`old()` 값과 에러 메시지가 서로 다른 폼에 섞여 들어가는 문제가 생겼다.

예를 들어 댓글 수정 실패 후 `old('content')`가
일반 댓글 작성 폼에까지 들어가는 식의 충돌이 발생했다.

처음에는 `editing_comment_id`, `replying_comment_id`를 기준으로
폼마다 예외적으로 분기하는 방식으로 해결했지만,
폼 수가 늘어날수록 관리가 복잡해질 수 있다는 점이 보였다.

정리한 방향:

- 댓글 작성: `comment_content`
- 댓글 수정: `edit_content`
- 답글 작성: `reply_content`

처럼 필드명을 역할별로 분리했다.

그리고 `CommentRequest` 안에 `commentField()` 헬퍼 메서드를 추가해
현재 요청이 어떤 폼에서 왔는지 판단한 뒤,
해당 필드만 trim, validation, error message 대상으로 삼도록 정리했다.

이 과정을 통해,
`하나의 화면 안에 여러 폼이 동시에 존재할 때는 필드명을 의도적으로 분리하는 편이 장기적으로 더 안정적`이라는 점을 정리했다.

## 16. `scopeBindings()`가 중첩 리소스 구조를 단순하게 만들어준 점

게시판, 게시글, 댓글처럼 URL이 중첩되는 구조에서는
처음에 컨트롤러 안에서 직접 소속 관계를 확인하는 코드가 자주 들어갔다.

예를 들면:

- 이 게시글이 정말 이 게시판 소속인지
- 이 댓글이 정말 이 게시글 소속인지

같은 검증을 수동으로 쓰게 되었다.

하지만 `scopeBindings()`를 적용한 뒤에는,
라우터가 먼저 부모-자식 관계를 기준으로 모델을 찾아주기 때문에
컨트롤러에서 중복 검증 코드를 많이 줄일 수 있었다.

이 경험을 통해,
중첩 리소스 구조를 사용할 때는 `scopeBindings()`를 먼저 검토하는 것이
Laravel스럽고 유지보수에도 더 유리하다는 점을 배웠다.

## 17. 임시 첨부 정리를 스케줄로 넘긴 이유

임시 첨부는 글 작성 도중 업로드되고, 실제 게시글에 연결되지 않은 채 남을 수 있다.
처음에는 업로드 자체만 구현해도 충분해 보였지만, 시간이 지나면 스토리지와 DB에 고아 데이터가 계속 쌓이는 문제가 생긴다.

최종적으로는 다음 구조로 정리했다.

- `attachments:cleanup-temporary` Artisan command 작성
- `routes/console.php`에 스케줄 등록
- `is_temporary = true` 이고 `post_id`가 없는 첨부 중 일정 시간이 지난 파일만 정리

이 과정을 통해,
실제 운영 관점에서는 `업로드 성공`만큼이나 `남은 임시 데이터 정리 정책`이 중요하다는 점을 정리했다.

## 18. 검색 UI는 정보 카드보다 목록 툴바가 더 자연스러웠다

게시판 내부 검색을 처음에는 `section-card`로 따로 감쌌지만,
검색은 새로운 정보 섹션이라기보다 게시글 목록을 조작하는 도구에 더 가까웠다.

결국 검색창은 목록 상단의 얇은 툴바로 옮기고,
필드 선택(`ID`, `제목`, `글쓴이`)과 검색어 입력을 같은 줄에서 다루는 형태로 정리했다.

이 경험을 통해,
같은 컴포넌트를 재사용하는 것보다 `이 UI가 실제로 어떤 역할을 하는가`를 먼저 보는 편이 더 자연스럽다는 점을 정리했다.

## 19. 홈 인기글은 Eloquent 결과를 뷰용 데이터로 한 번 더 정리하는 편이 깔끔했다

홈 인기글을 붙이면서 처음에는 `Post` 모델을 그대로 뷰에 넘기고, Blade 안에서 필요한 값을 계속 조합하는 방식으로 생각했다.
하지만 홈 카드에서 실제로 필요한 값은 `썸네일`, `제목`, `게시판명`, `작성 시각`, `댓글 수` 정도로 한정되어 있었다.

그래서 컨트롤러에서 `map()`으로 결과를 한 번 더 가공해,
뷰가 그대로 렌더링하기 좋은 형태의 배열로 정리하는 방식이 더 읽기 쉽고 부담이 적었다.

이 과정을 통해,
`쿼리 결과를 그대로 뷰에 넘기는 것`보다 `화면에 필요한 데이터 구조로 한 번 정리해서 넘기는 것`이 Blade 템플릿을 더 단순하게 만들어준다는 점을 배웠다.

## 20. 대표 이미지 1개를 뽑을 때는 `hasOne + oldestOfMany()`가 유용했다

게시글 본문에는 이미지가 여러 장 들어갈 수 있지만,
홈 인기글 카드에서는 그중 첫 번째 이미지만 썸네일처럼 보여주고 싶었다.

처음에는 첨부 전체를 가져온 뒤 PHP에서 첫 번째 이미지를 고르는 방식도 가능했지만,
이렇게 하면 관계 로딩과 화면 로직이 함께 복잡해질 수 있었다.

그래서 `Post` 모델에 `thumbnailAttachment()` 관계를 따로 만들고,
`hasOne()`에 `oldestOfMany()`를 조합해 "대표 첨부 1개"를 모델 관계 수준에서 표현하도록 정리했다.

이 경험을 통해,
Eloquent 관계는 단순한 목록 조회뿐 아니라 `대표 데이터 1개를 의미 있게 뽑아내는 용도`로도 충분히 활용할 수 있다는 점을 배웠다.

## 21. 인기글은 실시간 랭킹과 선정 이력을 분리하는 편이 더 자연스러웠다

처음에는 홈 인기글을 `좋아요 수`, `댓글 수`, `조회수` 기반 점수식으로 바로 정렬해서 보여주려고 했다.
하지만 이 방식은 "지금 점수가 높은 글"을 보여주기엔 편해도, 시간이 지나면 순위가 계속 바뀌고 `한 번 인기글로 선정된 글`이라는 개념을 남기기 어렵다는 문제가 있었다.

그래서 구조를 아래처럼 정리했다.

- 점수식은 `인기글 선정 여부`를 판단하는 기준으로만 사용
- 실제 인기글 목록은 `popular_posts` 테이블에 선정 이력으로 저장
- 홈 화면은 `PopularPost`를 기준으로 조회

이 경험을 통해,
실시간 랭킹과 선정 기록은 비슷해 보여도 성격이 다르기 때문에 `선정 이력을 따로 저장하는 구조`가 더 운영 친화적이라는 점을 배웠다.

## 22. 홈 인기글은 매 요청 계산보다 `Cache::remember()`가 더 현실적이었다

홈은 가장 자주 열리는 페이지라서, 인기글 계산 쿼리를 매 요청마다 직접 실행하는 방식은 금방 부담이 될 수 있다.
반면 인기글은 몇 초 단위로 완전 실시간일 필요는 없어서, 짧은 주기로 캐시해도 체감상 큰 문제가 없다.

최종적으로는:

- `PopularPost` 기준 인기글 조회
- 홈 컨트롤러에서 `Cache::remember()`로 1분 캐시
- 뷰에는 `map()`으로 가공한 최소 데이터만 전달

방식으로 정리했다.

이 경험을 통해,
홈처럼 조회가 많은 화면에서는 `조금 덜 실시간이어도 되는 데이터`를 캐시로 넘기는 편이 훨씬 현실적이라는 점을 다시 확인했다.

## 23. 상태 조건은 컨트롤러보다 모델 `scope`로 올리는 편이 훨씬 읽기 좋았다

프로젝트가 커지면서 컨트롤러마다 아래 같은 조건이 계속 반복되기 시작했다.

- `status = published`
- `status = visible`
- `is_active = true`
- `published post + active board`

처음에는 단순한 `where()`와 `whereHas()`로 충분했지만,
조건이 반복될수록 쿼리가 길어지고 "무슨 의미의 조건인지"를 매번 해석해야 하는 문제가 생겼다.

그래서 모델에 아래처럼 `local scope`를 정리했다.

- `Board::active()`
- `Post::published()`
- `Post::publishedOnActiveBoards()`
- `Comment::visible()`
- `PopularPost::available()`

이후 컨트롤러에서는 조건식을 길게 나열하는 대신,
`Post::published()`, `PopularPost::available()` 같은 식으로 바로 의미가 읽히는 쿼리를 사용할 수 있게 되었다.

이 과정을 통해,
단순히 쿼리를 짧게 만드는 것보다 `조건의 의미를 이름으로 드러내는 것`이 유지보수에 훨씬 큰 도움이 된다는 점을 체감했다.

## 24. 관계 데이터 전체가 아니라 존재 여부만 필요할 때는 `withExists()` / `loadExists()`가 더 잘 맞았다

처음에는 게시판 목록이나 게시글 상세에서
`popularEntry` 관계를 그대로 `with()`나 `load()`로 붙이는 쪽부터 떠올렸다.
하지만 실제로 필요한 것은 인기글 상세 정보 전체가 아니라 `이 글이 인기글로 선정되었는가` 같은 불리언 정보인 경우가 많았다.

이때 `withExists()`와 `loadExists()`를 사용하니,
관계 레코드 전체를 억지로 가져오지 않고도 `$post->popular_entry_exists`처럼 의도가 바로 읽히는 형태로 화면을 만들 수 있었다.

이 경험을 통해,
관계는 항상 전체 데이터를 로드하는 것보다 `지금 화면에 필요한 정보가 무엇인지` 먼저 생각하고
존재 여부만 필요하면 `withExists()` 계열을 쓰는 편이 더 깔끔하다는 점을 배웠다.

## 25. 문자열 `title` prop만으로는 제목 옆 배지 같은 UI를 자연스럽게 붙이기 어려웠다

`x-ui.section-card`는 처음에 `title` 문자열만 받는 단순한 구조였다.
이 구조는 일반적인 섹션 제목에는 충분했지만,
게시글 상세처럼 `인기` 배지와 제목을 같은 줄에서 함께 보여주고 싶은 상황에서는 바로 한계가 드러났다.

문자열을 억지로 조합하는 방식은 마크업을 섞기 어렵고,
제목 자체에 표시용 텍스트를 덧붙이는 쪽도 UI가 금방 어색해졌다.

그래서 `section-card`에 `heading` 슬롯을 추가해,
기본적으로는 `title`을 그대로 쓰되 필요한 곳에서는 `배지 + 제목` 마크업을 직접 넣을 수 있게 바꿨다.

이 경험을 통해,
공통 컴포넌트는 처음부터 과하게 복잡할 필요는 없지만
`문자열 prop만으로는 표현하기 어려운 UI`가 보이는 시점에는 슬롯으로 확장할 수 있도록 여지를 두는 편이 좋다는 점을 배웠다.

## 26. 반복되는 작은 배지도 결국 컴포넌트로 빼는 편이 더 좋았다

처음에는 `인기`, `공지`, `조회`, `추천`, `댓글` 같은 작은 배지를
각 화면에서 Tailwind 클래스로 직접 적는 쪽이 빠르게 느껴졌다.
하지만 인기글 페이지, 일반 게시판, 게시글 상세, 홈 공지처럼 사용처가 늘어나자
같은 톤의 마크업이 여러 파일에 반복되기 시작했다.

그래서 `x-ui.badge` 컴포넌트를 추가하고,
반복되는 배지류를 한 번에 같은 톤으로 바꾸는 방식으로 정리했다.

이 과정을 통해,
배지처럼 사소해 보이는 UI도 `반복 위치가 3곳 이상` 보이기 시작하면
그때가 공통 컴포넌트로 빼기 가장 좋은 시점이라는 점을 확인했다.

이 문서는 이후 기능을 구현하면서 계속 보완할 수 있다.
